1. プロダクトの核となる哲学

「人間がリアルタイムで分かること」と「AIが後処理で分かること」を分業させる。
最後にLLMが重み付きで統合 → 役割分担の明確化が品質と運用容易性を両立させる。
軸担当強み弱み
誰が話したか人間（録音中の話者ボタン押下）100%の正確さ・即時判断押し忘れ・遅延
何を話したかAI（Whisper / Deepgram / Web Speech）大量データ処理・固有名詞推定話者分離は苦手・誤字あり

4. 技術スタック一覧

フロントエンド

  • Vanilla JavaScript（約1,500行）
  • Service Worker / Manifest
  • MediaRecorder API（webm/opus 32kbps）
  • Web Speech API（ja-JP）
  • IndexedDB（ローカル退避）

バックエンド

  • FastAPI（Python）
  • SQLite（/var/jizo/jizo.db）
  • Nginx + Let's Encrypt
  • systemd: jizo-api（:8002）
  • ファイルローテーション（12h）

外部API

  • OpenAI Whisper API（whisper-1）
  • Deepgram API（nova-3 / nova-2）
  • Google Gemini API（2.5-flash / pro）
  • OpenAI互換エンドポイント経由

インフラ

  • VPS: Ubuntu 22.04（jizo-dev.com）
  • HTTPS必須（マイク/Speech API要件）
  • GitHub: kurapomu/ai-voice-memo-v1.1
  • PC管理画面: Basic認証

データストア

  • projects / recordings
  • ref_segments（WS参考）
  • speaker_events（押下記録）
  • ai_transcripts（Run1/Run2）
  • merged_transcripts（統合結果）

運用ルール

  • API Key は VPS .env のみ
  • SW更新時はCACHEバージョン昇格
  • 録音20分上限（Whisper 25MB制約）
  • STT無反応20秒で警告バナー
  • グラデーション全面禁止

5. 重み付き統合アルゴリズム（独自設計）

従来：LLMに3ソースを丸投げ → 勝手にマージ・省略・偏りが発生。
新設計：アルゴリズム側で「行構造」「話者」を確定 → LLMはテキスト選択のみに集中。
WS = Web Speech API Run1 = 1回目のSTT解析 Run2 = 2回目のSTT解析
1重みから骨組みソースを選定

例：WS=2, Run1=4, Run2=4 → 同点時は run2 > run1 > ws → 骨組み = Run2

2骨組みの全セグメントを行に展開

Run2 が 30 セグメントなら、出力も 30 行で固定。LLMが行数を変えることを構造的に禁止。

3各行に他ソースを「候補」として添える

骨組み行のタイムスタンプから ±2秒以内 の最近傍を採用。3ソースの候補テーブルが完成。

4話者をアルゴリズムで決定（LLMに任せない）

優先度チェーン：押下イベント±2秒 → 直前継承 → WS ref_segments → 未設定（【要確認:話者】 付記）

5LLMに構造化テーブルと重み比を渡す

指示は「行を追加・削除・マージするな。重み比に従って3候補から選ぶ／合成せよ」。LLMの仕事はテキスト合成のみ → 暴走しない。

候補テーブルの例（ステップ3の出力）

行01 [00:02] WS="こんにちは"   Run1="こんにちは"      Run2="こんにちは"
行02 [00:23] WS="ＤＳＲ"        Run1="DSL"            Run2="DSR について"
行03 [00:45] WS=(なし)          Run1="今日のアジェ"    Run2="今日のアジェンダは"

重み設定UI（管理画面イメージ）

┌────────────────────────────────────────┐
│ LLM統合の重み付け（合計10）             │
│                                          │
│   WS  2  :  Run1  4  :  Run2  4         │
│                                          │
│ Web Speech ●──────                  2   │
│ Run1       ────●──                  4   │
│ Run2       ────●──                  4   │
│                                          │
│ [Run2重視 1:2:7] [均等 2:4:4] [WS重視 7:2:1] │
└────────────────────────────────────────┘

スライダー1本を動かすと他2本が比例配分で自動調整（合計10維持）。重み 0 のソースは LLM 入力から完全除外。

6. 設計上の重要な制約

制約理由
createSTT() のみ差し替え可Web Speech依存を将来切り替えても、UI・録音・IndexedDBに影響を出さない
録音20分/チャンクWhisper API のファイルサイズ25MB制限
HTTPS必須Web Speech API とマイクアクセスの要件
Basic認証は管理画面のみモバイル側は誰でもアップロード可（軽い運用ポリシー）
APIキーは全部VPS側クライアント漏洩防止
グラデーション全面禁止ユーザー指定の絶対ルール
STT無反応20秒で警告iOS Safari でバックグラウンドStopする問題への対処
