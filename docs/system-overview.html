1. プロダクトの核となる哲学

「人間がリアルタイムで分かること」と「AIが後処理で分かること」を分業させる。
最後にLLMが重み付きで統合 → 役割分担の明確化が品質と運用容易性を両立させる。
軸担当強み弱み
誰が話したか人間（録音中の話者ボタン押下）100%の正確さ・即時判断押し忘れ・遅延
何を話したかAI（Whisper / Deepgram / Web Speech）大量データ処理・固有名詞推定話者分離は苦手・誤字あり

2. システム全体構成図

モバイルPWA・VPS・外部API・管理画面の4層に加え、Zoom Cloud Recording を OAuth で取り込む経路を持つ。データはすべて VPS の SQLite と音声ストレージに集約される。

Zoom連携は Pull型一覧フロー：管理画面の「Zoom録画一覧」モーダルから明示的に1件ずつ取り込み。Webhook は通知記録のみで自動DL はしない。

外部API

  • OpenAI Whisper API（whisper-1）
  • Deepgram API（nova-3 / nova-2）
  • Google Gemini API（2.5-flash / pro）

インフラ

  • VPS: Ubuntu 22.04（jizo-dev.com）
  • HTTPS必須（マイク/Speech API要件）
  • GitHub: kurapomu/ai-voice-memo-v1.1
  • PC管理画面: Basic認証

データストア

  • projects / recordings
  • ref_segments（WS参考）
  • speaker_events（押下記録）
  • ai_transcripts（Run1/Run2）

  • SW更新時はCACHEバージョン昇格

  • グラデーション全面禁止

5. LLM統合：ゼロ欠落・時系列整流アーキテクチャ（独自設計・2026-06-05改修）

旧設計の破綻：「重み最大ソースを骨組み → LLMが全行を一度に整形」設計は、出力トークン truncation、Gemini の指示不遵守、検証不在の3欠陥で「後半が欠損だらけ」になっていた。
新設計：行構成は完全に決定論的に確定し、LLM の仕事は行内テキストの整流のみに限定。チャンク分割 + 強制整合フォールバックで ゼロ欠落・時系列順・LLM障害でも結果が返る を構造で保証。
WS = Web Speech API Run1 = 1回目のSTT解析 Run2 = 2回目のSTT解析
1主軸選定（決定論）

運用デフォルトは fixed で Run1（Whisper）を主軸固定（日本語精度と分割安定性のため）。自動選定にも切替可能：rows_x_log_chars（行数 × log文字数）/ rows_only / chars_only。

2行構築（主軸 + オーファン挿入）

主軸の各行に他2ソースの最近傍候補を ±2秒窓で添付。主軸どの行とも窓外かつ広域類似度 < 0.4 の他ソース発話は「オーファン」として時系列で挿入（取りこぼし救済）。500ms以内の隣接オーファンは連結。ここで行数が確定し、以降一切変動しない。

3話者解決（決定論・LLMに任せない）

優先度チェーン：押下イベント±2秒 → 直前押下継承 → WS speaker_idx ±4秒 → 「未設定」（【要確認:話者】 付記）。

4チャンク分割LLM整流（並列）

全行を chunk_size=25 行ずつに分割、最大3並列で LLM 呼出。各チャンクに前後2行を「文脈」として付与。LLMの仕事は厳格に3つだけ：①欠損単語の補完 ②語順修正 ③整形。行追加・削除・マージ・並べ替え・候補に無い語句生成は絶対禁止。出力は JSONL（1行=1JSON、truncation 耐性が高い）。

5強制整合（決定論）

チャンクごとに：LLM 失敗 or 行数不一致 → そのチャンク全行を主軸生テキストで埋める。各行で「候補に無い4字以上連続部分文字列」を検出 → 幻覚と判定し主軸生テキストで上書き。LLM障害でも結果は必ず返る。

重みスライダーの新しい意味

旧設計で「主軸選定の重み」だった w_ws:w_run1:w_run2 は、新設計では LLM への補完優先度ヒント として渡される（同等候補から1つ選ぶ際の参考）。主軸選定は別の自動アルゴリズムが担当する。スライダー UI は localStorage に保存され、MTGをまたいで継続される。運用デフォルトは 1:7:2（主軸 Run1 固定との整合）。

マージ結果のレスポンス例

{
  "ok": true,
  "segments": 141,           // 確定行数（ゼロ欠落で保持）
  "notes": 0,                // 要確認件数
  "backbone": "run1",        // 自動選定された主軸
  "fallback_chunks": 1,      // LLM が壊れて決定論で補償したチャンク数
  "hallucinations": 0        // 幻覚検出して上書きした行数
}

fallback_chunks > 0 でも結果は完成している（決定論フォールバックで補償済み）。hallucinations > 0 の行は note=「要確認:LLM過剰補填可能性」 が付き、管理画面の要確認リストに集約される。
